# Это веб интерфейс для проекта "Анализатор погоды"
С помощью данного сайта можно
* Посмотреть, насколько точно разные сервисы предсказывают прогноз погоды
* Настроить подписки определённые города, чтобы получать самые свежие прогнозы

**Цель** - создать сайт, на котором можно будет посмотреть анализ точности сайтов прогноза погоды, а также управлять подписками на получение погодной информации по электронной почте.

**Целевая аудитория** - люди, чья повседневная жизнь, работа или хобби зависят от погоды. Это туристы, дачники, садоводы, спортсмены, организаторы мероприятий, курьеры, а также обычные городские жители, которые хотят доверять прогнозу, чтобы эффективно планировать свое время. 

**Ключевая потребность**, из-за которой мой сайт им будет полезен — получать достоверную информацию о погоде и понимать, каким источникам можно доверять.


# Возможности
## Функциональные
* Пользователь должен иметь возможность регистрироваться и входить в систему                                                                  
* Авторизованный пользователь должен иметь посмотреть аналитику точности предсказания температуры/осадков по всем 15 городам/по определённому городу. 
* Система должна показывать графики зависимости точности предсказания прогноза погоды от срока предсказания для всех 5 сервисов. Метрики: RMSE, боксплоты (для t); precision, recall (для осадков)
* Авторизованный пользователь должен иметь возможность настроить подписку на получение погодных данных по определённым городам
* Система должна отправлять письма с погодными данными в соответствии с настроенными подписками
* Логи пользователей должны сохраняться в БД
* Администраторы (пользователи с ролью ADMIN) должны иметь возможность посмотреть логи, в том числе отфильтровав по пользователю/типу
* Администраторы (пользователи с ролью ADMIN) должны иметь возможность управлять чужими подписками
## Нефункциональные
* Страница с графиками должна загружаться не более 5 секунд, остальные страницы - не более 2 секунд
* Интерфейс должен корректно отображаться на мобильных устройствах
* Все возникающие ошибки должны отображаться на странице в читаемом формате
* Статистика точности прогнозов (температура, осадки) должна быть представлена в виде понятных графиков
* Доступ к функционалу аналитики, управления подписками и личным данным должен быть только у авторизованных пользователей.
* Доступ к логам и управлению чужими подписками должен быть только у пользователей с ролью ADMIN.
* Пароли пользователей должны храниться в БД в зашифрованном виде
* Система должна быть защищена от SQL-инъекций, XSS, CSRF
# Концепция пользовательского интерфейса (UI/UX)
## Основные страницы:
* Страница входа
* Главная страница
* Страница со статистикой
* Страница с подписками
* Страница с логами (только для админов)
## Пользовательские сценарии:
#### Регистрация + настройка подписки
* пользователь заходит на сайт, 
* нажимает кнопку “зарегистрироваться”,
* регистрируется (вводит свой email, придумывает логин и пароль), 
* заходит на страницу подписок,
* подключает подписку на погоду в Сочи и Калуге (нажимает на кнопку “+” рядом с названиями соответствующих городов)
#### Авторизация + просмотр аналитики
* Пользователь авторизуется на сайте (нажимает на кнопку “войти”, вводит логин и пароль, нажимает на кнопку “ОК”), 
* система отправляет запрос /login на backend, получает авторизационный токен и перенаправляет пользователя на главную страницу, 
* пользователь заходит на страницу аналитики,
* выбирает в раскрывающемся списке “RMSE” и “Москва”,
* система делает запрос /stat на backend и получает нужный график и выводит его на страницу,
* пользователь смотрит график зависимости RMSE прогноза температуры в Москве от срока предсказания 
#### Отключение подписки
* Пользователь заходит на страницу подписок,
* отключает подписку на погоду в Калуге (нажимает кнопку “Отписаться” рядом с названием соответствующего города), 
* система отправляет DELETE-запрос на backend и выводит на экран новый список подписок
#### Просмотр логов
* Администратор авторизуется на сайте (вводит свой логин и пароль),
* система отправляет запрос /login на backend, получает авторизационный токен и перенаправляет админа на главную страницу, 
* он заходит на страницу логов,
* в фильтре вводит имя пользователя “hacker”, система отправляет запрос /logs, выводит на экран полученный список логов,
* администратор просматривает их
# Архитектура системы
### Frontend: Vue.js (для динамичного интерфейса).
### Backend: Flask (обрабатывает запросы, хранит данные).
### База данных: PostgreSQL (хранит пользователей, города, подписки, логи)
